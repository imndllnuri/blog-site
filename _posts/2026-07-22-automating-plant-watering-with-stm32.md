Watering a plant reliably is about as simple as control problems get: read
a sensor, compare it to a threshold, actuate a pump. That simplicity is
exactly why I picked it — it let me practice the *full* embedded workflow
end to end, from circuit design through HAL configuration through
simulation-before-hardware validation, without the actual control logic
being complicated enough to distract from the process. Wiring a sensor to a
dev board and reading a value off a serial monitor teaches you a lot less
than building the whole closed loop.

The result is a functional hardware prototype around an STM32F103C
(Cortex-M3): soil-moisture and water-level sensors read via ADC/DMA, both
readings shown live on a character LCD, and a pump/relay driven
automatically once soil moisture drops below a threshold.

## Simulate the whole circuit before touching hardware

Before any soldering happened, the full circuit ran in Proteus —
STM32F103C6, soil-moisture probe, water-level sensor, and the character LCD
all simulated together, running the exact firmware STM32CubeMX generated.

![Proteus circuit simulation showing live sensor readings on the LCD](/assets/img/projects/stm32-automated-plant-watering/proteus-simulation.png)
_The full circuit running in Proteus: STM32F103C6, soil moisture probe, water-level sensor, and character LCD showing live readings (`Soil: 3979`, `Water: 459`) before any hardware was touched._

That's not a cosmetic step. Simulating first caught pin-mapping and
threshold-logic mistakes in software, where getting something wrong costs a
re-simulation instead of a re-soldering session. By the time real hardware
came into the picture, "bring-up" was mostly a formality rather than the
first real debugging pass — Proteus animating the actual generated firmware
against a simulated circuit meant most of the debugging had already
happened before a single wire was connected for real.

## Two ADC channels, and why DMA instead of polling

The sensor side is two ADC channels: `ADC_CHANNEL_8` reads the soil-moisture
probe, `ADC_CHANNEL_9` reads the water-level sensor, both set up via
`HAL_ADC_ConfigChannel`. The obvious way to read them is to block on
`HAL_ADC_PollForConversion` in the main loop — and the obvious way is the
wrong one here, because a single-threaded `while(1)` that blocks on a
conversion ties up the CPU for the whole read, right when the same loop
also needs to keep the LCD responsive and re-check the pump threshold
promptly.

Instead, `MX_DMA_Init()` wires up `DMA1_Channel1` so ADC conversions
complete in the background — the main loop just calls `ReadSensors()` then
`UpdateLCD()` in a tight cycle, free to service the display and the
threshold check without ever stalling on a conversion. It's a small
architectural choice, but it's the difference between a control loop that
stays responsive and one that silently misses a threshold crossing while
it's busy waiting on the ADC.

## Letting CubeMX handle the boilerplate

The peripheral setup — clock trees, GPIO/ADC/DMA configuration — is
generated by STM32CubeMX rather than hand-written against the F1-series
register map. That's not laziness so much as scoping: CubeMX-generated HAL
initialization is correct and portable, and hand-rolling it for an F1 part
is tedious and easy to get subtly wrong in ways that are hard to debug
later. Letting CubeMX own that layer meant the actual project-specific work
— reading sensors, applying the threshold, actuating the pump — could sit
cleanly on top instead of being buried in clock-configuration boilerplate.

The threshold check itself is the simplest part of the whole system: once
soil moisture drops past the configured value, a GPIO pin toggles the pump
relay. That simplicity was the point from the start — everything more
complex in this project is in the surrounding workflow (simulate, validate,
then build), not in the control logic itself.
